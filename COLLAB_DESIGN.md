# コラボ字幕 設計メモ（ROADMAP #4）

コラボで自分＋相手の字幕を別々のボックスに出す。2案あり、排他ではなく**入口として並存**させる。

- **案A改（1対1・Discordループバック話者分離）**: ホスト1台だけで完結。相手はノータッチ。
  → 本ドキュメント「案A改」節（下）
- **案B（各自ローカル・リレー共有）**: 全員がMojicast勢・多人数・最高精度の本命。
  → 本ドキュメント「案B」節（以降の大半）

使い分け:

| | 案A改 | 案B |
|---|---|---|
| 相手の準備 | **不要**（スマホ/ゲーム機コラボもOK） | 各自Mojicast導入 |
| 人数 | **1対1限定** | 多人数可 |
| 精度 | 中（Discord圧縮音声） | 最高（クリーンなローカルマイク） |
| ホスト負荷 | ASR2本＝ほぼ倍 | 各自1本に分散 |
| インフラ | 不要（ホスト内完結） | Cloudflareリレー |
| 今すぐ度 | 高（仮想ケーブルで今日試せる） | 中（リレー実装が要る） |

---

# 案B（各自がMojicastを起動・確定字幕テキストを共有）

各参加者が自分のPCで Mojicast を起動し、**確定字幕テキストだけ**をリレー経由で
共有して、全員分の字幕を各自のOBSに別々のスタイルで表示する。

音声は一切送らない。各自のクリーンなマイク音声をローカル認識するため、
精度は最高・話者の帰属は完璧・負荷は各PCに分散・帯域はテキストのみでほぼゼロ。

---

## 1. アーキテクチャ

```
[えも PC]                                     [ゲスト PC]
  マイク → Mojicast（ローカル認識）             マイク → Mojicast（ローカル認識）
     │ 確定字幕を送信                              │ 確定字幕を送信
     ▼                                           ▼
  ┌────────────── リレー（Cloudflare Workers + Durable Objects）─────────────┐
  │  部屋コードごとに1つの「部屋」。受け取ったメッセージを部屋の全員へ配るだけ  │
  └──────────────────────────┬───────────────────────────────────────────┘
     │ 全員分の字幕(WebSocket)                     │ 全員分の字幕
     ▼                                           ▼
  OBS オーバーレイ（話者ごとにスタイル/ボックス）   OBS オーバーレイ（同じく）
```

### リレーの技術詳細
- **Cloudflare Worker**: WebSocketへのアップグレードを受け、部屋コードから
  対応する Durable Object へルーティングするだけの薄い層
- **Durable Object（部屋）**: 接続中クライアントの一覧を持ち、受信メッセージを
  全員へブロードキャストする。数十〜百行規模
  - WebSocket Hibernation API を使い、待機中の課金・リソースを最小化
  - 字幕は**保存しない**（素通しのみ・ログ無効）。部屋は全員切断で消滅
- **メッセージ形式**（JSON・1行1メッセージ）:
  ```json
  { "type": "final", "speaker": "えも", "fid": 123,
    "text": "こんばんは", "ts": 1234567890 }
  { "type": "translation", "speaker": "えも", "fid": 123, "text": "Good evening" }
  { "type": "presence", "members": ["えも", "ゲストA"] }
  ```
  - 英訳は各自のPCが自分の発話分を翻訳して後追い送信（負荷分散・fidで対応付け）
  - **スタイル共有**: 参加時（と変更時）に自分の使用中プリセットを送る
    ```json
    { "type": "style", "speaker": "ゲストA",
      "style": { …presets.json の1項目と同じ形… },
      "effects": [ …エフェクト単語（任意）… ] }
    ```
    受信側は「その話者の字幕」をこのスタイルで描画（mojipackと同じデータ形式・
    数KB程度）。ローカルの presets.json には保存しない一時適用
- **部屋コード**: 推測困難なランダム8文字程度をホスト側で生成。TLS必須（wss://）
- **コスト**: テキストのみ・数人・配信時間分の接続なので Cloudflare 無料枠で十分
  - 試算: 2人・3時間・英訳ONのコラボ1回 ≒ 4,000〜5,000メッセージ。
    無料枠（約10万リクエスト/日）で最も保守的に見ても **1日20〜30コラボ**、
    Hibernationの換算（メッセージ20件=1リクエスト）が効けば **1日数百コラボ**
  - 課金ラインは「日常的に数百コラボ/日」＝数千人規模のコミュニティ。
    そこまで育っても Workers 有料プランは **月$5**（1,000万リクエスト/月）
  - 無料プランは上限到達で**止まるだけで勝手に課金されない**
    （リレーが止まってもローカル字幕は無影響。翌日復活）
  - ※Hibernationの換算比率など細かい課金仕様は実装時に最新の料金表で要確認
- 遅延はリレー数十ms程度で、認識処理（数百ms）に比べ無視できる

### 代替案（リレー無し・最初の一歩に）
Cloudflare Tunnel / Tailscale でホストのMojicast APIへゲストが直接POSTする構成なら
サーバーデプロイ不要。ただし「全員のOBSに全員分を表示」にはリレー型が必要なので、
本命はリレー型（この代替はホストのOBSにだけ出す1画面コラボ用）。

## 2. Mojicast側の改修（実装タスク）

| # | 内容 | 規模 |
|---|---|---|
| 1 | リレーWorkerの実装・デプロイ（wrangler） | 小（数十〜百行） |
| 2 | 送信: 確定字幕/英訳をWSでリレーへ（コラボON時のみ） | 小 |
| 3 | 受信: リレーからのメッセージを既存SSEパイプラインへ合流 | 中 |
| 4 | overlay: 話者→スタイル/ボックスの割当て表示（resolve_styleの拡張） | 中 |
| 4b | スタイル共有: 話者の見た目＝本人が送ったプリセットで描画。<br>文字スタイルは話者の自己表現・**ボックス配置は画面の持ち主が決める**という分担。<br>受信スタイルは名前等を無害化（mojipackインポートと同じ処理）し、<br>フォントは名前参照＋無ければフォールバック（同梱不可のため）。<br>「相手のスタイルを使う/自分で割り当てる」を話者ごとに切替可 | 小〜中 |
| 5 | コックピットUI: コラボグループ（表示名・部屋の作成/参加・メンバー表示） | 中 |
| 6 | 再接続・エラー処理（リレー断でもローカル字幕は止めない） | 小 |

合計 2〜3日級。v1は**確定字幕のみ**共有（認識中の薄文字は将来拡張。
帯域的には可能だが、まず表示がシンプルな確定のみで様子見）。

## 3. セキュリティ / プライバシー

- 字幕テキストがCloudflareを経由する（TLS・保存なし）。この点は使う人に明示する
- 部屋コードを知っている人しか入れない（コードは毎回使い捨て）
- 受信テキストはoverlayで textContent 描画（既存実装のままHTML注入不可）
- 表示名も無害化してから表示

## 4. ユーザーが準備する手順

### 開発者（えもさん）の一度きりの準備
1. Cloudflareアカウント作成（無料。メールだけでOK）
2. `wrangler`（Cloudflare CLI）でリレーWorkerをデプロイ
   （コードはリポジトリに同梱予定。コマンド2つ程度）
3. 発行されたリレーURL（`wss://…workers.dev/…`）をMojicastの既定値に設定
→ 以降、テスターやコラボ相手は**このリレーをそのまま使える**（各自のデプロイ不要）

### 重要: 「部屋を立てる」にアカウントは不要
部屋の作成＝アプリがランダムな部屋コードを生成するだけ。部屋は共用リレー上に
自動的に生まれる（Durable Objectは部屋コードごとに自動インスタンス化）ため、
**一般ユーザーはCloudflareアカウントも設定も一切不要**。
Cloudflareアカウントが要るのは「リレーを自前でデプロイする人」（＝開発者、
または自前リレー派の上級者）だけ。

### ホスト（部屋を作る人）
1. Mojicast を起動 → コックピットの「コラボ」で表示名を入れて **部屋を作成**
2. 発行された**部屋コードをDiscord等で相手に共有**
3. いつも通り ▶開始。OBSの設定は変更不要（自分のoverlayに全員分が出る）

### ゲスト（参加する人）
1. 各自のPCに Mojicast をインストール（無料・通常のセットアップと同じ）
   - PCがMojicastの動作要件を満たしていること（コラボによる追加負荷はほぼ無し）
2. 「コラボ」で表示名＋**部屋コードを入力して参加**
3. 自分のマイクを選んで ▶開始
4. 自分も配信するなら、自分のOBSにいつも通りoverlayを設定（全員分が出る）

### 前提・制約
- 参加者全員がWindows PC＋マイク＋Mojicast動作要件を満たすこと
- 初回はモデルDL（約1.2GB）が各自必要
- スマホ勢・Mojicastを入れられない相手は対象外（その場合は案A=ループバックで
  「相手まとめて1ボックス」が代替）

## 5. 決めごと（実装前に確定したい）

- [ ] 部屋の同時人数上限（案: 8人）
- [ ] スタイル共有の既定（案: 相手のスタイルをそのまま使う＝ON。話者ごとにOFFにして
      自分のプリセットへ割当も可能）
- [ ] エフェクト単語も共有するか（本人の「ありがとう」が相手の画面でも光る。
      案: スタイルと一緒に送る・受信側でOFF可）
- [ ] リレーURLをユーザーが差し替え可能にするか（自前リレー派向け・案: configに持つ）
- [ ] 認識中（薄文字）の共有は将来拡張とする（v1は確定のみ）

# 案A改（1対1・Discordループバック話者分離）

ホストPCだけで自分と相手の2人分の字幕を出す。**相手はMojicast不要**。
ROADMAPの旧「案A」（相手まとめて1ボックス）を、**話者分離までできる**形に格上げしたもの。

肝は「相手の声を独立した第2ストリームとして取り込む」こと。物理的に別ストリームなので
**話者の帰属は原理的に完璧**（マイク＝自分 / Discord出力＝相手、混ざりようがない）。

## 1. アーキテクチャ

```
[ホスト（えも）PC]
  ① マイク          ──→ Mojicast ASR #1 ──→ 話者「えも」
  ② Discord出力だけ ──→ Mojicast ASR #2 ──→ 話者「ゲスト」
                              │ 2話者を既存パイプラインへ合流
                              ▼
                     OBS オーバーレイ（話者ごとに別スタイル/ボックス）
```

## 2. ②「Discordの音だけ」を分離する2方式

### 方式1: 仮想オーディオケーブル（まず動かす検証用）
- VB-CABLE 等の仮想デバイスを入れ、Discordの**出力先**をその仮想デバイスに向ける
- Mojicastは仮想デバイスを**入力デバイス**として掴む（既存のデバイス選択UIで選ぶだけ）
- 長所: アプリ側の実装は「入力を2つ選んで別話者に割当」だけ。**今日試せる**
- 短所: ユーザーが1回セットアップ（仮想ケーブル導入＋Discord出力先変更）が要る。
  相手の声だけを仮想ケーブルに送り、自分は通常デバイスでモニタする配線に注意

### 方式2: WASAPIプロセスループバック（本命・内製化）
- Windows 10 2004（build 19041）以降の**プロセス指定ループバック**で
  「Discord.exe の出力音声だけ」をOS機能で直接キャプチャ。仮想ケーブル不要
- `pyaudiowpatch`（WASAPIループバック対応のPyAudioフォーク）等で実装
- 長所: ユーザー設定ゼロ。「Discordを選ぶ」だけで済む理想形
- 短所: 実装がやや込み入る（プロセス選択UI・対象プロセスの解決）。
  方式1で有用性を確認してから着手

## 3. 制約・落とし穴（正直なところ）

- **1対1限定は本質的**: Discordに3人以上いると全員の声が②の1ミックスに混ざり分離不可。
  話者分離ASRなしでは原理的に無理 → 「1対1コラボに限り」で正しい
- **精度低下**: Discord音声は圧縮＋相手側のノイズ抑制済み。クリーンなマイク直より落ちる（案B優位）
- **ホスト負荷ほぼ倍**: ASRを2本同時に回す。最低スペックの引き上げ／要件チェックが要る
- **ヘッドホン必須**: スピーカー出力だとマイク①がDiscordの音を拾い二重字幕（エコー）になる。
  ヘッドホンなら①に相手の声は入らず問題なし
- **自分の声の戻りは無し**: Discordは自分のマイクをループバックしないので②に自分は入らない

## 4. Mojicast側の改修（実装タスク）

| # | 内容 | 規模 |
|---|---|---|
| 1 | 音声入力を**2ストリーム同時**に走らせられるようにする（現状の単一入力を一般化） | 中 |
| 2 | 各ストリームに**固定話者ラベル**を付与（①=自分 / ②=相手） | 小 |
| 3 | ASR結果を話者付きで既存パイプライン（SSE/overlay）へ合流 | 小（案Bの受信合流と共通化可） |
| 4 | overlay: 話者→スタイル/ボックス割当（`resolve_style`拡張・**案Bと同じ仕組みを共用**） | 中 |
| 5 | コックピットUI: 「1対1コラボ（出力キャプチャ）」モード＝②の入力デバイス選択＋話者名 | 中 |
| 6 | （方式2のみ）プロセス指定ループバックの実装・対象アプリ選択UI | 中〜大 |

案Bと**話者付き描画（#3・#4）は共通基盤**にできるので、先にどちらを作っても他方が楽になる。
v1は方式1（仮想ケーブル）＋確定字幕のみで最小検証。

## 5. 決めごと（実装前に確定したい）

- [ ] 最初の実装は方式1（仮想ケーブル）でよいか（有用性確認 → 方式2内製化の順）
- [ ] 2ストリーム同時ASRの負荷が最低スペックで許容内か（要実測）
- [ ] ②の話者名の既定（案: 「ゲスト」。UIで変更可）
- [ ] 案Bの話者付き描画基盤と共通化する設計にするか（案: する）
