## 正式版（v1.0）の判断基準

ベータ運用（v0.4系〜）で以下を満たしたら正式版として公開する:

1. **実戦フィードバックの反映** — テスターと自分の常用配信で出た改善点を消化しきる
2. **UI/UXのさらなる最適化** — 迷いなく使える導線（初見のオンボーディング・設定の
   分かりやすさ・エラー時の案内）を突き詰める
3. **CPU差の吸収** — /api/perf の実測収集 → スレッド自動チューニング・
   自動パフォーマンスモード等で「適性◎〜△のCPUなら設定いじらずそれなりに動く」状態に
   （候補メニュー: ORT更新 / コラボ時ゲスト薄文字オフ / SenseVoiceハイブリッド。
   Intel/AMD別ビルドは最終手段）
4. **配布形態の完成** — Inno Setupインストーラ化（下記#2）・Booth出品（無料＋応援版、
   応援版特典は限定mojipack案）

## 2. Inno Setup インストーラ化
**規模: 小〜中（≈1日）**

- 配布物が setup.exe 1個になり、展開の遅さ・MOTW・フォルダの威圧感が構造的に解消
- v0.3.0のONNX/CT2移行で 約1,500ファイル/0.43GB・Zip 178MB まで軽くなったため
  緊急度は下がったが、テスターの導入体験としては依然価値あり
- onefile化は非推奨（この規模は起動毎のtemp展開が遅すぎる）

## 3. 別ソフト「話題レーダー」（仮称）
**規模: PoC半日 / 本体2〜3日 / パッケージ肥大回避のため別アプリ**

小型ローカルLLMで「今の話題」を単語抽出し、OBSにふわっと浮かべるコンパニオンアプリ。

- Mojicast側は**改修ゼロ**: SSE `http://localhost:8765/events` で確定字幕を購読できる
- 構成: 直近3〜5分のローリングウィンドウ → 30〜60秒毎にLLMで話題語3つ →
  自前ポートでOBS用オーバーレイ配信
- ランタイム: llama.cpp / GGUF（torch不要・数十MB）＋ Q4量子化モデル（初回DL方式）
- **モデル本命: Qwen3-1.7B**（CPU実試験で実用性確認済み・2026-07-12）
- CPU/GPU切替: `n_gpu_layers`（0=CPU / -1=全GPU）。バックエンドは **Vulkan推奨**
  （NVIDIA/AMD/Intel対応・同梱サイズ小。CUDAは数百MB膨らむ）
- **既定はCPU**（ゲーム配信中のGPUはゲーム+NVENCで激務）。
  「GPUを使う」トグル既定OFF＋GPU初期化失敗時はCPUフォールバック。
  雑談配信・2PC構成ではGPUが快適
- CPU配慮: 低スレッド・低優先度のバースト処理でASRを邪魔しない
- 発展: 話題語を `/api/effects` に流して字幕内を自動ハイライト（API既存）

## 4. コラボ字幕・案B（各自ローカル＋リレー共有・多人数対応）
**規模: 中〜大 / 詳細設計は [COLLAB_DESIGN.md](COLLAB_DESIGN.md) / 案A改（1対1）は v0.4.0 で実装済み**

各自がMojicastを起動し、**確定字幕テキストだけ**をリレー共有する多人数対応の本命構成。

- 各自のクリーンなマイク音声でローカルASR → 精度最高・話者帰属完璧・負荷分散。
  共有するのはテキストのみで帯域ほぼゼロ・多人数可（v0.4.0の1対1制限を超える）
- リレー: Cloudflare Workers + Durable Objects の小さなWebSocketリレー
  （部屋コード方式・無料枠で十分。数十行規模）
- 素朴な代替: Cloudflare Tunnel / Tailscale でホストのAPIへゲストが直接POST
  （サーバー運用ゼロ・ホストのOBSに全員分表示）
- 受信側の改修は小さい: 話者付き描画基盤（話者→スタイル/ボックス割当・overlay複数ペイン）は
  **v0.4.0 で実装済み**なので、リモート字幕を既存パイプラインへ流すだけ
- スタイル共有（相手の見た目=本人のプリセット）は COLLAB_DESIGN.md の #4b 参照

## 6. 多言語対応（モデル切替の「受け皿」）
**規模: 中 / 主要部は実装済み（2026-07-22・feat/asr-multilingual→v0.5.0候補）**

Twitterで中国語圏からの要望（2026-07-21）を起点に、ASR・翻訳をモデル選択式にした。

- **実装済み**: モデルレジストリ受け皿（k2/SenseVoice＋能力表→engineが後処理を自動切替）・
  SenseVoice多言語認識（中英日韓粤・軽量モード兼用: 13700HXで翻訳込み13〜20%、
  N100で条件付き動作）・翻訳4方向（FuguMT＋M2M-100、認識言語×翻訳先で経路自動選択・
  広東語はzh扱い）・読み欄「／」複数形の置換強化・AIモデル設定窓・
  初回おすすめ設定（OS言語/CPU検出・ユーザー了解制）・不要モデルの自動解放
- **残り**: 韓国語訳の品質改善（現状「試験的」表記）・多言語運用の実戦フィードバック消化
- 将来オプション: 翻訳品質の上位枠に MADLAD-400 3B（Apache-2.0・メモリ3GB級・
  16GB機向け）。NLLBは CC-BY-NC のため不採用
- **言語別ユーザー辞書（組み込み配信用語のユーザー拡充）**: 現状「英訳を固定する単語」＝
  ja→英のユーザー辞書は、組み込み `_STREAM_TERMS`（translate.py）と同じ「翻訳前の
  source側置換」機構で動く。これを中国語・インドネシア語など**M2M方向にも開放**する案
  （言語別 glossary を translate_m2m の事前置換へ）。
  - 壁: ja→英はラテン文字passthroughで堅牢だが、M2Mは注入した訳語を再翻訳して壊す語がある
    （実測: 歌回→歌曲回归・点赞→赞赞）。ユーザー任意登録には**登録時プレビュー/壊れ検証**が要る
  - 保存はプロファイル単位にし、#1 の mojipack単語系共有に載せれば「配信用語パック」を配れる
- 感情・音声イベントの活用は #7 参照

## 7. 音声由来のインタラクション（構想メモ・寝かせ枠 2026-07-22）
**規模: 感情エフェクト=小〜中 / ボイスコマンド=中（誤発動設計が本体）**

- **感情・音声イベント→エフェクト**: SenseVoiceは認識と同時に感情（喜怒哀楽）と
  音声イベント（笑い・拍手・咳等）のトークンを出せる（sherpa-onnxの結果に露出済み）。
  「笑い→(笑)＋パーティクル」「HAPPY→字幕がポップに」等のエモーショナル字幕。
  単語エフェクトの既存基盤（spawnParticles / effects API）にそのまま乗る。
  制約: 多言語モデル選択時のみ・感情判定は発話単位でそこそこの精度
- **ボイスコマンド**: 確定字幕のテキストマッチで「フレーズ→アクション」
  （クリア・翻訳オン・エフェクト発射・プロファイル切替等）。実装自体は小さいが、
  設計の本体は誤発動対策（呼びかけ語プレフィックス必須・コマンド文は字幕に出さない・
  確定ベースなので数秒の遅延がある前提の用途選び）
- どちらも #8 カスタムエフェクトと組み合わせると化ける（例: 笑い検出で自作エフェクト発動）

## 8. カスタムパーティクル / カスタムエフェクト
**規模: パーティクル=小 / エフェクト段階1=中 / 遠い将来枠**

ユーザーが「飛ばすもの」と「エフェクト」を自作できるようにする。
実装順は **パーティクル → エフェクト段階1 → 段階2**（小さい成功体験から積む）。

- **カスタムパーティクル（小）**: 実体は fx.js の `PARTICLE_DEFS`（絵文字・個数・
  飛び方・回転のJSON）なので、単語スタジオに4項目の小フォームを足すだけ。
  絵文字ならフォント同梱問題もなし
- **エフェクト段階1: レシピ型（中）**: 既存プリミティブのパラメータ開放
  （アニメ周期・グラデ色・グロー幅など）＋名前を付けて保存。
  `--fx-grad` CSS変数化済みでグラデ色の下地はあり。実体がJSONなので mojipack と好相性
- **エフェクト段階2: キーフレーム開放**: transform/opacity/filter の時系列JSONから
  `@keyframes` を生成（構造化データからCSSを組み立てる＝注入リスクを段階1と同等に維持）
- **生CSS/JSプラグインは開けない**（mojipack共有時のセキュリティホールになるため）
